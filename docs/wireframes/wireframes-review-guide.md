## 3. How To Read The Wireframes

The wireframes are intentionally low fidelity.

They define:

- structure
- content hierarchy
- object relationships
- shared component reuse
- desktop/mobile intent

They do **not** define:

- final styling
- exact spacing
- final copy polish
- interaction animation

## 4. Notation

- `[ComponentName]` means a reusable component or canonical block
- `[P]` means primary action
- `[S]` means secondary action
- `OBJECT:` identifies the main UX object on the screen
- `MODE:` identifies whether the screen is student, tutor, or shared

## 5. What To Review First

### Highest-value questions

1. Does the student journey feel guided rather than marketplace-like?
2. Do tutor operational screens still feel like the same product?
3. Are lessons, messages, and schedule visibly part of one shared ecosystem?
4. Is the tutor side framed as teaching workflow rather than admin back-office?
5. Are the primary actions obvious on each screen?

### Most important screens

If you only review a few screens first, review these:

1. Student Match Flow
2. Student Results / Search
3. Tutor Profile
4. Booking Flow
5. Tutor Overview
6. Tutor Lessons
7. Tutor Schedule

## 6. Expected Outcome Of Review

After reviewing the wireframes, we should be able to decide:

- whether the structure is right
- whether any screens are missing
- whether navigation labels should change
- whether any objects need to be split or merged
- whether we are ready to move into higher-fidelity layout and visual design

## 7. Next Step After Review

Once this wireframe pack is approved, the next design pass should be:

1. revise the wireframes based on feedback
2. define the shared component inventory and reuse matrix
3. create mid-fi screens for the core journey
4. then move into the visual design system and hi-fi work
